# Спецификация проекта: Бот для лиги

Единый источник правды по функционалу, правилам и текущему состоянию кода. При новом чате прочитай этот документ целиком — в нём весь контекст проекта.

**Суть:** Telegram-бот для одной хоккейной лиги: регистрация персон, роли (игрок/вратарь/работник/офицер), три независимых расписания (игроки, работники, вратари), уведомления, зарплата по наличным 5000₽ с группами и переводами, штрафы (5 минусов), рейтинги. Сейчас в работе — админ игроков и сценарии для игроков.

---

## Текущее состояние реализации

### Сделано

- **Регистрация и заявки:** пользователь вводит фамилию/имя → заявка админу; админ подтверждает/отклоняет, назначает одну или две роли (игрок, работник, вратарь, офицер). Бан = блокировка доступа.
- **После одобрения:** приветственное сообщение + предложение установить фон чата (кнопки «Фон чата РХЛ» — ссылка на обои, «Пропустить»). Меню пользователя приходит по «Пропустить» сразу или автоматически через 3 сек (если открыл ссылку или ничего не нажал). Клавиатура фона: `bot/keyboards/user/wallpaper.py`.
- **Админ игроков:** панель (Расписание, Управление, Личное и т.д.). Создание расписания на дату: ввод даты, времени, числа игр, составов 1–3 команд с валидацией по БД; можно добавлять несколько туров в один день («Добавить тур» сохраняет текущий тур в список и начинает следующий). Итоговый экран: «Опубликовать» или «Редактировать игровой день» (список туров → выбор тура → правка времени/игр/составов). Удаление расписания запрещено.
- **Редактирование прошлого расписания:** «Редактировать расписание» → ввод даты → загрузка туров из БД → тот же сценарий редактирования тура → сохранение обновляет `Tour` и пересчитывает `PlayerTourStats` (зарплатные данные актуальны). Если отредактированный день — завтра, игрокам уходит короткое уведомление об изменениях.
- **Публикация расписания:** по «Опубликовать» — сохранение в БД и рассылка **полного** расписания на этот день всем активным игрокам (развёрнутый формат). Рассылка и короткое уведомление «завтра изменилось» живут в `bot/services/schedule_notifications.py`; список получателей — `get_active_players_telegram_ids()` в `db/crud.py`.
- **Даты:** для тестирования разрешено создавать расписание задним числом (проверка «не в прошлом» отключена).
- **БД:** SQLite, async SQLAlchemy. Модели: `Person`, `DateTour`, `Tour`, `PlayerTourStats`, `WorkerSchedule`. Ключевые функции: `get_tours_by_date_tour_id`, `update_tour`, `save_schedule_to_db`, `get_active_players_telegram_ids`; при нескольких игроках с одной фамилией для сохранения туров берётся первый (`scalars().first()`).

### Не сделано / в планах

- Просмотр расписания пользователем (кнопка «Расписание»), роутер пользователя.
- Зарплата: расчёт за период, наличные 5000₽, группы, минимизация переводов, персональные инструкции.
- Штрафы (5 минусов, офицеры, модерация), рейтинги.
- Расписание работников и вратарей, админы работников и вратарей, главный админ.

## Технический стек и структура

- **Стек:** Python, aiogram (Telegram Bot API), SQLAlchemy (async), SQLite, pydantic-settings для конфига.
- **Конфиг:** `bot/config.py` — `BotSettings` (bot_token, admin_players, admin_worker, admin_goalkeeper и др.). `.env` для секретов. Опционально: `ANCHOR_MONDAY=YYYY-MM-DD` — якорный понедельник для двухнедельных периодов (рейтинги, зарплата); если не задан, используется понедельник текущей недели (Калининград).
- **Роутеры:** `bot/main.py` подключает registration, admin (approval, players, schedule). Обработчики расписания админа — `bot/handlers/admin/schedule.py` (объёмный файл: FSM создания туров, редактирование черновика и БД, публикация).
- **БД:** `db/session.py` (async engine/session), `db/models.py`, `db/crud.py`. Миграции не используются — создание таблиц при старте.
- **Уведомления и рассылка расписания:** `bot/services/schedule_notifications.py` — `send_full_schedule_to_players`, `notify_players_schedule_changed` (только если дата = завтра), `build_player_schedule_message`.
- **Клавиатуры:** `bot/keyboards/admin/` (approval, schedule, admin_player), `bot/keyboards/user/` (main, registration, wallpaper). Состояния FSM — `bot/states/schedule.py`, `bot/states/registration.py`.

## 1. Масштаб проекта

- **Одна конкретная лига** — мультилига/смена лиги не планируется.

## 2. Роли персон и расписания

### Три типа расписания (независимые)

- **Расписание игроков** — ведёт админ игроков.
- **Расписание работников** — ведёт админ работников (оператор, директор, К-центр, комментатор, судья и т.д.).
- **Расписание вратарей** — ведёт админ вратарей.

Это **три разных расписания**, а не одно с разными фильтрами.

### Общие функции (для всех ролей)

- Штрафы (просмотр своих; выставление — только офицеры).
- Рейтинги (просмотр).
- Перечень общих функций может расширяться по ходу работы — уточняем в процессе.

### Роли персон

| Роль      | Расписание                    | Примечание |
|-----------|-------------------------------|------------|
| Игрок     | только расписание игроков     | |
| Работник  | только расписание работников  | |
| Вратарь   | только расписание вратарей    | |
| Игрок + работник | оба: игроков и работников | два блока в уведомлениях/разделе |
| Вратарь + работник | расписание вратарей + расписание работников | два блока |

**Офицер** — дополнительная роль к админам или к пользователям. Не заменяет игрока/работника/вратаря. Даёт право выставлять штрафы по лиге.

- Лимита на количество ролей у одного человека **нет**.

---

## 3. Админы

- **4 роли в перспективе:** 1 главный админ + 3 зональных (админ игроков, админ работников, админ вратарей).
- **Главный админ** — роль есть в БД, реализация в конце проекта.
- **Зональные админы:** каждый редактирует **только своё** расписание:
  - админ игроков — только расписание игроков;
  - админ работников — только расписание работников;
  - админ вратарей — только расписание вратарей.

---

## 4. Редактирование расписания

- Редактировать расписание в своей зоне может **только соответствующий админ**.
- **Удалять** записи расписания **нельзя** — по ним считается зарплата (история должна сохраняться).

---

## 5. Кто что видит по расписанию

- **Игрок** — только расписание игроков (развёрнутый формат).
- **Работник** — только расписание работников (развёрнутый формат).
- **Вратарь** — только расписание вратарей (развёрнутый формат).
- **Вратарь + работник** — два расписания: вратарское и рабочее (оба развёрнутые).
- **Игрок + работник** — расписание игроков + расписание работников.

Формат уведомлений/экран расписания — **развёрнутый** (дата, время, состав, при необходимости место и т.д.).

---

## 6. Штрафы

- Выставляют только **офицеры** (в т.ч. админ, если он офицер).
- Офицер выставляет штрафы **по всей лиге** (не привязан к «своей» зоне).
- **Модель:** система из **5 минусов**.
  - За каждое нарушение — 1 минус.
  - При **5 минусах за месяц** пользователь обязан оплатить штраф.
- **Модерация:** админ может модерировать штрафы (подтверждать/отменять — уточнить при реализации).

---

## 7. Зарплата

### Общие условия

- В организации зарплата только **наличными**, купюрами по **5000 ₽**.
- Расчёт ведётся по данным расписания (игры/смены за период).
- Принцип один и тот же для **игроков**, **работников** и **вратарей** (отдельные расчёты по зонам).

### Схема для игроков (аналогично — для работников и вратарей)

1. **Период расчёта**  
   Админ вводит период, например: с 10.02.26 по 24.02.26.

2. **Расчёт суммы**  
   - Бот по расписанию суммирует игры каждого игрока за период.  
   - Умножает на ставку за одну игру.  
   - Получается сумма к выдаче по каждому и общая требуемая сумма (напр. 1 324 540 ₽).

3. **Наличные у админа**  
   Когда у админа появляются наличные, он вводит сумму (напр. 1 325 000 ₽).  
   Бот:
   - показывает требуемую сумму (1 324 540);
   - показывает, сколько наличных учтено;
   - предлагает сумму, **кратную 5000**, для «ровного» распределения.

4. **Выравнивание**  
   Разница между наличными и суммой к выдаче компенсируется **переводами**:
   - либо игроки переводят админу (излишки);
   - либо админ переводит игрокам (недостача).

5. **Группы и минимизация переводов**  
   - Бот делит пользователей с ролью игрока на **группы по 4–5 человек**.  
   - Цель: админ передаёт каждой группе наличные одной «пачкой», при этом **минимизировать количество переводов** между людьми (внутри групп и между группами, включая переводы с админом).

6. **Инструкции**  
   - **Общая** — по группам (кто в какой группе, сколько наличными по группе и т.д.).  
   - **Персональная** каждому игроку:
     - в какой группе он находится;
     - сколько получить наличными;
     - от кого получить перевод или кому перевести излишки (чтобы в итоге у всех было рубль к рублю);
     - в сообщении указывается **username** для удобства.
   - Переводы могут быть: внутри группы, между группами, с админом.

### Пример персональной инструкции (игрок)

Игроку приходит:
- общая инструкция по группам;
- персональная инструкция: твоя группа, сумма наличными, «оставшуюся сумму получить переводом от @username» или «перевести излишки @username».

## 10. Приоритеты реализации

1. **Админ игроков до финала** — приём заявок, роли, создание и редактирование расписания игроков (без удаления), отправка расписания игрокам. *(В основном реализовано; остаётся довести сценарии пользователя.)*
2. **Пользователи с ролью игрок** — просмотр расписания игроков, уведомления в развёрнутом формате.
3. **Зарплата** — расчёт по периоду, наличные/кратное 5000, разбиение на группы, минимизация переводов, персональные и общие инструкции (игроки; затем по тому же принципу работники и вратари).
4. **Позже:** штрафы (5 минусов, офицеры, модерация), рейтинги, расписание работников и вратарей, админы работников и вратарей, главный админ.

## 11. Демо и ограничения

- Реализуется только **админ игроков** (остальные админы и главный — позже).
- Расписание игроков: создание (в т.ч. несколько туров в день), редактирование черновика и прошлых дней (без удаления), публикация с рассылкой полного расписания игрокам; при изменении завтрашнего дня — короткое уведомление.
- Для тестирования разрешено создавать расписание на прошедшие даты.
- После одобрения заявки пользователю предлагается фон чата (ссылка на обои); меню приходит по «Пропустить» или через 3 сек.
