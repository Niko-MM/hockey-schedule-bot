## Этап 1. Админ игроков до финала

**Цель:** админ игроков полностью закрывает цикл: заявки → роли → расписание (создание + редактирование в истории) → рассылка игрокам.

### 1.1 Уже сделано (без изменений)

- [x] Регистрация персоны (фамилия, имя) → заявка админу
- [x] Подтверждение/отклонение заявок, выбор ролей (игрок, работник, вратарь, офицер), вторая роль
- [x] Админ-панель: меню, раздел «Расписание»
- [x] Создание расписания на дату: туры, время, игры, составы 1–3 команд, валидация по БД, сохранение в БД и статистика игр (`PlayerTourStats`)

### 1.2 Редактирование расписания в истории

- [ ] **CRUD:** получить туры по дню — `get_tours_by_date_tour_id(date_tour_id) -> list[Tour]` (сортировка по времени).
- [ ] **CRUD:** обновить тур — `update_tour(tour_id, ...)` (время, игры, составы) без удаления записи.
- [ ] Админ: пункт «✏️ Редактировать расписание» → выбор даты из существующих дней (`DateTour`) → список туров на этот день → выбор тура → редактирование (аналогично текущему FSM: время, игры, составы команд) → сохранение. Без удаления дня/тура.
- [ ] Убрать заглушку в `handle_edit_schedule` (сейчас «будет в следующем обновлении»).

### 1.3 Рассылка расписания игрокам

- [x] После «Опубликовать» (confirm_publish): кроме сохранения в БД — собирать список активных игроков с `telegram_id` (роль игрок) и отправлять каждому развёрнутое сообщение с расписанием на эту дату (дата, туры, время, составы).
- [x] Учесть, что в демо в БД могут быть тестовые персоны без реального Telegram — ошибки отправки игнорируются для конкретного пользователя, общий поток не падает.
- [x] Текст сообщения: развёрнутый формат (как в спецификации п.5).

**Критерий завершения этапа 1:** админ может принять заявку, выставить роли, создать расписание, отредактировать его по дате и отправить игрокам; удаление расписания недоступно.

## Этап 2. Пользователи с ролью игрок

**Цель:** игрок видит расписание и получает уведомления в развёрнутом формате.

### 2.1 Роутер и меню пользователя

- [x] Подключены роутеры `bot/handlers/user/schedule.py`, `bot/handlers/user/salary.py`; обрабатываются кнопки из `get_user_keyboard()` (📅 Расписание, 📅 Выбрать дату, Штрафы, Рейтинги, Зарплата, Правила).
- [x] Меню выдаётся после одобрения заявки (всем с хотя бы одной ролью).

### 2.2 Просмотр расписания игроков

- [x] **CRUD:** `get_last_schedule_dates(limit=10)` — последние даты с расписанием.
- [x] «📅 Расписание» — показ расписания на ближайший игровой день (развёрнутый формат).
- [x] «📅 Выбрать дату» — бот присылает текстом список последних 10 дат в формате ДД.ММ.ГГ и подсказку ввести дату; пользователь вводит ДД.ММ.ГГ → расписание на выбранную дату. Формат сообщения единый с рассылкой.

### 2.3 Уведомления при публикации

- [x] При публикации админом игроки получают полное расписание; при изменении завтрашнего дня — короткое уведомление. Список получателей — персоны с ролью игрок.

**Критерий завершения этапа 2:** выполнен.

## Этап 3. Зарплата (игроки)

**Цель:** расчёт зарплаты игроков за период, учёт наличных 5000₽, разбиение на группы, минимизация переводов, персональные и общие инструкции.

### 3.1 Расчёт за период

- [ ] Админ вводит период (дата начала, дата конца). Бот считает по `PlayerTourStats` + `Tour` + `DateTour` игры каждого игрока в периоде, умножает на `player_rate` → сумма к выдаче по каждому и общая.
- [ ] Вывод админу: общая требуемая сумма; при необходимости — список по игрокам.

### 3.2 Наличные и кратность 5000

- [ ] Админ вводит сумму наличных (сколько пришло). Бот показывает: требуемая сумма, введённая сумма, предлагает сумму кратную 5000 для «ровного» распределения (сверх требуемой или с возвратом админу).
- [ ] Логика выбора кратной суммы: минимизировать остаток/добавку при разбиении на группы (уточнить при реализации).

### 3.3 Группы и минимизация переводов

- [ ] Алгоритм разбиения игроков на группы по 4–5 человек и расчёт, кто кому переводит (внутри группы, между группами, с админом), чтобы количество переводов было минимальным, а итог у каждого — рубль к рублю.
- [ ] Учёт username для указания в инструкциях («получить перевод от @username» / «перевести @username»).

### 3.4 Инструкции

- [ ] Общая инструкция админу: группы, суммы наличными по группам, кому передать пачки.
- [ ] Персональное сообщение каждому игроку: твоя группа, сумма наличными, от кого получить перевод / кому перевести излишки (с @username). Отправка в Telegram.

**Критерий завершения этапа 3:** админ задаёт период и получает расчёт; вводит наличные → получает разбиение на группы и кратную 5000 сумму; игроки получают персональные инструкции с переводами и username.
